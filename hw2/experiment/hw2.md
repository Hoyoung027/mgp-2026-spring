# GEMV / GEMM 성능 최적화 기법

## GEMV 최적화 기법

### 1. 멀티스레딩 — 행(row) 단위 병렬 분할

**원리**: N×N 행렬의 각 행은 서로 독립적으로 계산될 수 있습니다. 2048개 행을 T개 스레드에 균등 분할하여 각 스레드가 자기 구간 `[start, end)`만 담당합니다. 공유 자원은 읽기 전용인 `a[]`, `b[]`뿐이고 출력 `c[]`는 스레드 간 겹치지 않아 동기화 없이 병렬 실행이 가능합니다.

```
스레드0: 행 0~255
스레드1: 행 256~511
...  (T개 스레드 동시 실행)
```

### 2. 4-way 스칼라 Unrolling + 다중 누산기 (ILP)

**원리**: 하나의 `sum`에 순차 누적하면 각 반복이 이전 결과에 데이터 의존성을 가져 CPU 파이프라인이 막힙니다. 누산기를 `s0~s3` 4개로 분리하면 의존성이 끊어져 CPU가 여러 연산을 동시에 파이프라인에 투입(Instruction-Level Parallelism)할 수 있습니다.

```cpp
// 단일 누산기: 매 반복이 이전 값에 의존 → 파이프라인 정지
sum += row[j] * b[j];

// 4개 누산기: 서로 독립 → 파이프라인 병렬 실행
s0 += pr0[0] * bj0;
s1 += pr0[1] * bj1;   // s0와 독립
s2 += pr0[2] * bj2;   // s0, s1과 독립
s3 += pr0[3] * bj3;   // 모두 독립
```

### 3. 2행 동시 계산 — b[] 메모리 재사용

**원리**: 행 1개를 계산하려면 b[] 전체(16KB)를 읽어야 합니다. 행 2개를 동시에 계산하면 동일한 `b[j]` 값을 두 행에 재사용할 수 있어 b[]에 대한 메모리 접근이 절반으로 줄어듭니다. `-O0`에서도 로컬 변수 `bj0~bj3`는 스택(= L1 캐시)에 올라가므로 실제 메모리 재접근이 방지됩니다.

```cpp
double bj0 = pb[0];          // b[j] 메모리에서 1회만 로드
s0 += pr0[0] * bj0;          // row0에 사용
t0 += pr1[0] * bj0;          // row1에도 재사용 (추가 메모리 접근 없음)
```

### 4. 포인터 증가 패턴 — 인덱스 연산 제거

**원리**: 배열 인덱싱 `row0[j]`는 내부적으로 `*(row0 + j)`로, 매 반복마다 덧셈 연산이 발생합니다. 포인터를 직접 증가시키면 이 인덱스 계산 자체가 사라집니다. `-O0`에서는 컴파일러가 이 최적화를 자동으로 하지 않으므로 수동 적용 효과가 큽니다.

```cpp
// 변경 전: 매 반복 row0 + j 계산
row0[j], row0[j+1], row0[j+2], row0[j+3]

// 변경 후: 컴파일 타임 상수 오프셋 + 포인터 이동
pr0[0], pr0[1], pr0[2], pr0[3]  →  pr0 += 4
```

외부 루프도 동일하게 적용:

```cpp
// 변경 전: 매 외부 반복마다 곱셈
double *row0 = a + i * N;    // 곱셈 발생

// 변경 후: 덧셈만
row0 += 2 * N;               // 곱셈 없음
```

### 5. 루프 종료 조건 사전 계산

**원리**: `b + N - 3`을 `b_end`로 사전 계산해두면 내부 루프 매 반복마다 이 값을 재계산하지 않습니다. `-O0`에서는 컴파일러가 루프 불변 값을 자동으로 끌어내지 않으므로 효과가 있습니다.

```cpp
const double *b_end = b + N - 3;  // 1회 계산
for (; pb < b_end; ...)            // 매 반복: 비교만
```

## GEMM 최적화 기법

### 1. 멀티스레딩 — 행 단위 병렬 분할

GEMV와 동일 원리. 출력 행렬 c[]의 행 구간을 스레드별로 분할해 독립 계산합니다.

### 2. ikj 루프 순서 — 캐시 친화적 접근

**원리**: 행렬은 row-major(행 우선) 순서로 메모리에 저장됩니다. 루프 순서에 따라 캐시 미스 횟수가 크게 달라집니다.

```
ijk 순서: b[k*N+j] 접근 시 k가 변할 때마다 행이 바뀜 → cache miss 빈번
ikj 순서: b[k*N+j]에서 j만 변함 → 연속 메모리 접근 → cache friendly
```

`aik = a[i*N+k]`를 내부 루프 밖으로 끌어내 반복 접근도 제거합니다.

### 3. 4-way Unrolling — 루프 오버헤드 감소

**원리**: j 루프를 4단계씩 전개해 루프 분기/증감 횟수를 1/4로 줄이고, `b[k*N+j]` 연속 접근으로 캐시 라인을 효율적으로 활용합니다.

```cpp
// 1회 루프 반복에 4개 연산
c[i*N+j]   += aik * b[k*N+j];
c[i*N+j+1] += aik * b[k*N+j+1];
c[i*N+j+2] += aik * b[k*N+j+2];
c[i*N+j+3] += aik * b[k*N+j+3];
```

## 한계

컴파일 플래그 없이 기본 `-O0`으로 빌드되므로 컴파일러 자동 최적화(레지스터 할당, 벡터화)가 동작하지 않습니다. 위 기법들은 모두 이 제약 하에서 코드 레벨로 수동 적용한 것이며, `-O2` 환경 대비 여전히 큰 오버헤드가 남아 있습니다.
