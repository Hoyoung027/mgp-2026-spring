# 1부: AI 작성 페이지

## 1. 병렬 알고리즘 구현

`parallel-double.h`에는 `gemv`(행렬-벡터 곱), `gemm`(행렬-행렬 곱), `init_vec`(벡터 초기화) 세 가지 핵심 루틴이 구현되어 있다. OpenMP 사용이 금지되어 있으므로 모든 병렬화는 `std::thread`를 통해 구현하였다.

### gemv

행렬의 행(row)을 T개의 스레드에 분배한다 (T = min(hardware_concurrency, 8)). 각 스레드는 서로 다른 출력 벡터 원소에 쓰기 때문에 동기화가 필요하지 않다.

스레드 내부에는 두 가지 최적화 계층이 적용된다.

**2행 동시 계산**: 인접한 두 행(`row0`, `row1`)을 동시에 처리한다. 내부 루프에서 `b[j]`를 한 번만 로드하여 두 행 모두에 재사용함으로써 `b`에 대한 메모리 접근 횟수를 절반으로 줄인다.

**SSE2 8-언롤**: `__m128d`(2개의 double을 담는 128비트 SSE2 레지스터)를 사용하여 행마다 4개의 어큐뮬레이터를 유지한다. 총 8개의 `__m128d` 레지스터(`s0~s3`: row0용, `t0~t3`: row1용)가 사용되며, 한 이터레이션에서 `b`의 연속된 8개 원소를 처리한다. 메인 루프 종료 후 horizontal sum으로 4개의 `__m128d` 어큐뮬레이터를 스칼라로 변환한다.

```
__m128d rs = (s0+s1) + (s2+s3);
c[i] = _mm_cvtsd_f64(_mm_add_sd(rs, _mm_unpackhi_pd(rs, rs)));
```

N이 8의 배수가 아닌 경우를 위한 스칼라 나머지 루프도 존재하나, N=2048은 8의 배수이므로 실제로는 실행되지 않는다.

### gemm

출력 행렬 `c`의 행을 T개 스레드에 분배한다 (T = min(hardware_concurrency, 32)). 루프 순서는 **i → k → j** (row-major 친화적):

- `(i, k)` 쌍마다 `a[i][k]`를 `_mm_set1_pd`로 `__m128d` 레지스터에 브로드캐스트한다.
- j 루프에서 `_mm_loadu_pd` / `_mm_storeu_pd`를 사용하여 `b`와 `c`의 연속된 2개 열을 한 번에 처리한다.

이 접근 패턴은 `b[k][j..j+1]`과 `c[i][j..j+1]` 모두를 연속 메모리에서 접근하게 하여 캐시 라인 활용을 극대화한다.

### init_vec

고정 시드(`srand(42)`)로 결정론적 초기화를 보장한다. 각 원소는 `rand() / RAND_MAX`로 설정되어 [0, 1) 범위의 값을 가진다. 병렬화할 경우 `rand()`의 스레드 안전성 문제가 생기므로 순차 실행을 유지한다.

## 2. 어느 방식이 더 나은가

| 기준 | 병렬 GEMM | Freivalds 알고리즘 |
|---|---|---|
| 시간 복잡도 | O(N³) | O(N²) / 1회 |
| 결과 | 정확 | 확률적 |
| 소요 시간 (N=2048) | ~수 초 | ~수 밀리초 |
| 메모리 | O(N²) 출력 | O(N) 추가 |
| 용도 | C를 직접 계산 | 기존 C 검증 |

검증 목적에서는 Freivalds 알고리즘이 **압도적으로 빠르다**. N=2048 기준으로 GEMM은 약 86억 번의 곱셈-덧셈이 필요한 반면, Freivalds는 두 번의 `gemv` 호출(약 800만 번 연산)로 끝난다. 속도 차이는 약 O(N)배, 즉 N=2048에서 약 2000배에 달한다.

단, GEMM은 결과 행렬 자체가 필요한 경우에는 피할 수 없다. Freivalds는 "A×B = C인가?"라는 질문에만 답할 수 있다.

**결론**: 검증에는 Freivalds, 계산에는 GEMM이 필요하다.

## 3. Freivalds 알고리즘이 확률적인 이유와 오류 경계

Freivalds 알고리즘은 무작위로 선택한 벡터 v에 대해 A×(B×v) = C×v가 성립하는지 검사함으로써 A×B = C인지 확인한다.

**왜 확률적인가?** A×B ≠ C라고 하면 D = A×B − C ≠ 0이다. 검사를 통과하는 오탐(false negative)은 D×v = 0, 즉 v가 D의 영공간(null space)에 속할 때만 발생한다. {0,1}^N에서 무작위로 선택된 v에 대해:

> Pr[D×v = 0] ≤ 1/2

이는 Schwartz-Zippel 보조정리에 의한 결과로, 영이 아닌 행렬 D에 대해 이진 벡터 중 절반 이하만이 영공간에 속한다.

**오류 경계**: 서로 다른 무작위 벡터로 k번 독립 반복 시:

> Pr[k번 후에도 false negative] ≤ (1/2)^k

k=1: 오류 확률 ≤ 50%, k=20: ≤ 0.0001%. 실제 오류가 존재하면 대부분의 벡터에서 탐지되므로, 공학적 검증에서는 1회 시행으로도 충분한 경우가 많다.

본 구현에서는 `init_vec`이 고정 시드를 사용하므로 사실상 1회 시행만 수행된다.

# 2부: 학생 평가 페이지

## AI 내용 평가

### 올바른 내용

AI가 설명한 구현 내용은 전반적으로 정확하다. 스레드 모델(행 분배, 동기화 불필요), SSE2 최적화(8개 `__m128d` 어큐뮬레이터, 2행 동시 처리), horizontal sum 패턴 모두 실제 `parallel-double.h` 코드를 정확히 설명한다.

복잡도 분석(O(N³) vs O(N²))과 검증 목적에서 Freivalds가 빠르다는 결론도 맞다. N=2048에서 연산 횟수 비율이 실제로 약 N = 2048배임도 정확하다.

Schwartz-Zippel 기반의 오류 경계 도출(라운드당 ≤ 1/2, k회 시 ≤ (1/2)^k)도 올바르다.

### 틀리거나 부정확한 내용

**SSE2 효과 과장**: AI는 SSE2가 의미 있는 처리량 향상을 제공한다고 암시한다. 그러나 실제로 `gemv`는 메모리 바운드 연산이다. N=2048에서 `gemv` 한 번 호출 시 약 2048×2048×8 = 32MB의 행렬 데이터를 읽어야 한다. 일반적인 서버의 메모리 대역폭(~40 GB/s)을 감안하면 이 데이터 전송만으로 이미 ~0.8ms가 소요되며, SIMD 폭과 무관하게 이 한계를 벗어날 수 없다. `__m128d`의 이론적 2배 이득은 메모리 지연에 의해 상당 부분 상쇄된다.

**gemm j 루프 언롤**: AI는 gemm에서 이터레이션당 2개의 double을 처리한다고 설명하지만, 기존 스칼라 버전은 이미 4-언롤이었다. SSE 버전은 언롤 인수를 4에서 2로 줄인 것이다(다만 각 스텝이 2개 원소를 처리하므로 실제 처리량은 동일). 이 부분의 설명이 다소 불명확하다.

**고정 시드와 확률적 보장**: AI는 고정 시드를 사용하더라도 "행렬 값의 예측 불가능성에서 무작위성이 온다"고 설명한다. 그러나 이는 유효한 확률론적 논거가 아니다. 고정된 벡터는 무작위가 아니며, 학술적 의미에서 이 구현은 Freivalds의 확률적 보장을 충족하지 않는다. 엄밀한 구현은 매 실행마다 다른 무작위 벡터를 사용해야 한다.

### 빠진 내용

**스레드 수 제한 이유**: AI는 gemv가 8 스레드, gemm이 32 스레드로 제한되는 이유를 설명하지 않는다. gemv에서는 각 스레드가 전체 벡터 b를 읽으므로 8 스레드 수준에서 이미 메모리 대역폭이 포화된다. gemm은 스레드당 작업량이 더 크고 데이터 재사용이 가능해 더 많은 스레드가 유효하다.

**gemm의 캐시 효과**: i→k→j 루프 순서가 캐시 친화적으로 선택되었다는 설명은 있지만, `b[k][j]` 접근은 순차적(좋음)인 반면 k를 증가시킬 때의 `a[i][k]` 접근은 stride-N(잠재적으로 캐시 비친화적)이라는 점이 언급되지 않았다.

### 배운 점

가장 중요한 통찰은 **컴퓨트 바운드**와 **메모리 바운드** 작업의 구분이다. SIMD 최적화는 연산이 병목일 때 가장 효과적이다. `gemv`처럼 대용량 데이터를 읽는 작업에서는 CPU가 처리할 수 있는 속도보다 데이터를 공급하는 속도가 더 느리므로, 아무리 많은 곱셈을 한 사이클에 처리해도 한계가 있다.

Freivalds의 확률적 특성은 미묘하다. 고정 시드 1회 시험으로도 실용적으로는 충분하지만, 이론적 보장을 위해서는 진정한 무작위성이 필요하다는 점을 이 과제를 통해 명확히 이해하게 되었다.

## 6. 확률적 알고리즘이 유리한 상황

Freivalds 알고리즘은 확실성을 포기하는 대신 속도를 얻는 **랜덤화 알고리즘**의 초기 사례다. 이 패턴은 현대 시스템에서 널리 활용된다.

- **블룸 필터(Bloom filter)**: O(1)의 확률적 집합 소속 판별, 조정 가능한 위양성률
- **HyperLogLog**: 고정 메모리로 집합의 카디널리티 근사 계산
- **몬테카를로 방법**: 고차원 공간에서의 적분 근사

공통점은 정확한 계산이 O(N^k) 비용을 요구하는 반면, 확률적 검증은 제어 가능한 오류 허용 하에 O(N) 또는 O(N log N)으로 해결할 수 있다는 것이다. 행렬 검증 문제에서 Freivalds의 속도 이점은 문제 크기 N에 비례하여 증가하므로, 행렬이 커질수록 Freivalds가 더욱 유리해진다.

이 과제는 올바른 알고리즘 선택이 SIMD나 멀티스레딩 같은 저수준 최적화보다 훨씬 큰 성능 차이를 만들 수 있음을 보여준다. Freivalds의 ~2000배 속도 이점은 어떤 하드웨어 수준의 GEMM 최적화로도 달성할 수 없다.
