# HW5 - CUDA Sum Reduction 보고서

## 1. 과제 목표

이번 과제의 목표는 CUDA를 이용해 정수 배열의 합을 병렬 reduction으로 계산하는 것이다. 입력 배열은 이미 device memory의 `d_idata`에 복사되어 있으며, 구현한 GPU reduction 결과는 최종적으로 `d_odata[0]`에 저장되어야 한다.

강의자료의 Mark Harris CUDA reduction 예제를 바탕으로 `reduce0`부터 `reduce6`까지 총 7개의 kernel을 구현했다. 각 버전의 성능을 비교하기 위해 `reduce_optimize()`에서 호출하는 kernel을 바꿔가며 실험했고, 최종 제출용으로는 실험에서 가장 빠르게 측정된 `reduce5`를 선택했다.

## 2. 구현 내용

| Kernel | 강의자료 버전 | 최적화 내용 |
|---|---|---|
| `reduce0` | Reduction #1 | Interleaved addressing과 divergent branching 사용 |
| `reduce1` | Reduction #2 | Strided indexing으로 modulo 연산 제거 |
| `reduce2` | Reduction #3 | Sequential addressing 적용 |
| `reduce3` | Reduction #4 | Global memory load 단계에서 첫 번째 덧셈 수행 |
| `reduce4` | Reduction #5 | 마지막 warp 구간 unrolling |
| `reduce5` | Reduction #6 | Template block size를 이용한 complete unrolling |
| `reduce6` | Reduction #7 | Thread당 여러 원소를 grid-stride 방식으로 누적 |

구현에서는 block size를 256으로 설정했다.

```cpp
#define REDUCE_BLOCK_SIZE 256
#define REDUCE_MAX_BLOCKS 64
```

`reduce_optimize()`는 한 번의 kernel launch로 전체 배열을 바로 하나의 값으로 줄이지 않는다. 먼저 입력 배열을 block별 partial sum으로 줄이고, partial sum 배열에 대해 같은 reduction kernel을 반복 실행하여 최종적으로 하나의 값만 남긴다. 이 과정에서 최종 결과가 `d_odata[0]`에 저장된다.

## 3. 실험 환경 및 방법

실험은 `server8`에서 진행했다. 입력 크기는 Makefile에 정의된 네 가지 명령어를 사용했다.

| 명령어 | 입력 크기 |
|---|---:|
| `make 1` | 64 |
| `make 2` | 4,194,304 |
| `make 3` | 8,388,608 |
| `make 4` | 16,777,216 |

각 kernel 버전에 대해 `make 1`부터 `make 4`까지 실행하고, 출력된 `reduce_optimal took ... msec` 값을 기록했다. 대부분의 실험은 각 입력 크기마다 5회씩 수행했다.

특이사항은 다음과 같다.

- `reduce0` 실험 중 `make 4`를 실행해야 하는 구간에서 `make 3`가 한 번 잘못 실행되었다. 해당 값인 `0.562080 ms`는 `reduce0`의 `make 3` 통계에서 제외했다.
- `reduce4`의 `make 1`은 6회 실행되었고, 6개 값을 모두 포함했다.
- `reduce5`의 `make 4`는 이상치 확인을 위해 6회 실행했다. 이 중 `0.591520 ms`는 다른 다섯 값에 비해 크게 높아 이상치로 제외했다.

제공된 driver 기준으로 모든 실험에서 `Correct`가 출력되었다.

## 4. 전체 성능 비교

다음 표는 채점 기준 입력 크기인 `make 4`, 즉 `16,777,216`개 원소에 대한 성능 비교이다.

| Kernel | 최소 시간 (ms) | 최대 시간 (ms) | 평균 시간 (ms) | 비고 |
|---|---:|---:|---:|---|
| `reduce0` | 0.729216 | 0.845312 | 0.793901 | 기본 interleaved addressing |
| `reduce1` | 0.634336 | 0.759456 | 0.698432 | modulo 연산 제거 |
| `reduce2` | 0.609888 | 0.739616 | 0.647392 | sequential addressing |
| `reduce3` | 0.512320 | 0.541312 | 0.522605 | first add during global load |
| `reduce4` | 0.474624 | 0.607712 | 0.545965 | last warp unrolling |
| `reduce5` | 0.457856 | 0.477184 | 0.470054 | complete unrolling, 이상치 제외 |
| `reduce6` | 0.560512 | 0.697600 | 0.600851 | multiple elements per thread |

이번 실험에서는 `reduce5`가 평균 시간과 최대 시간 기준 모두 가장 빠르게 측정되었다. `reduce5`의 `make 4` 최대 시간은 `0.477184 ms`로, 과제 기준인 `1.2 ms`보다 충분히 낮다.

## 6. 분석

입력 크기가 64인 `make 1`에서는 실제 reduction 연산량이 매우 작기 때문에 kernel launch overhead가 실행 시간의 대부분을 차지한다. 따라서 모든 kernel이 약 0.27~0.30 ms 수준으로 비슷하게 측정되었다.

입력 크기가 커질수록 각 최적화 단계의 효과가 더 뚜렷하게 나타났다. `reduce0`은 가장 기본적인 interleaved addressing 구현으로 divergent branching과 modulo 연산 비용이 있다. `reduce1`은 modulo 연산을 제거하여 index 계산 비용을 줄였다. `reduce2`는 sequential addressing을 적용해 shared memory 접근 패턴을 개선했다. `reduce3`는 global memory load 단계에서 두 원소를 먼저 더해 block 수를 줄였다. `reduce4`는 마지막 warp를 unroll하여 synchronization overhead를 줄였고, `reduce5`는 reduction 단계를 complete unrolling하여 loop overhead를 더 줄였다.

강의자료의 최종 버전은 `reduce6`이며, 각 thread가 여러 원소를 처리하도록 하여 work per thread를 늘리는 방식이다. 하지만 이번 실험 환경에서는 `reduce5`가 `reduce6`보다 더 빠르게 측정되었다. `reduce6`에서 사용한 최대 block 수 제한과 grid-stride loop의 추가 제어 흐름이 현재 입력 크기와 서버 환경에서는 `reduce5`보다 유리하지 않았던 것으로 보인다.

## 7. 결론

`reduce0`부터 `reduce6`까지 모든 kernel이 제공된 driver에서 `Correct`를 출력했다. 또한 모든 버전이 채점 기준 입력 크기인 `16,777,216`개 원소에 대해 `1.2 ms` 미만의 실행 시간을 기록했다.

실험 결과 가장 빠른 버전은 `reduce5`였다. 따라서 최종 제출용 `reduce_optimize()`에서는 `reduce5`를 사용하도록 선택했다. 최종 선택 버전의 `make 4` 최대 시간은 이상치를 제외했을 때 `0.477184 ms`이며, 이는 과제 기준을 충분히 만족한다.
